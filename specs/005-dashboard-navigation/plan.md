# Implementation Plan: Dashboard Navigation Shell

**Branch**: `005-dashboard-navigation` | **Date**: 2026-07-01 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/005-dashboard-navigation/spec.md`

---

## Summary

Introduce a shared navigation shell for the authenticated dashboard with two destinations —
"Catálogo" (existing `/dashboard`, unchanged) and "Squads" (new `/dashboard/squads`, placeholder
content only). On desktop (`lg:` and up) the nav renders as a persistent left sidebar; below that
breakpoint it collapses into a hamburger toggle that opens an overlay drawer (dismissible via
backdrop tap or Escape). The nav is implemented as a new `app/(private)/dashboard/layout.tsx` so
both routes share it automatically, and lives entirely in client-side UI state — no new server
state, no new squad-context actions. Separately, a CSS-only reorder fixes the existing mobile bug
where the squad-in-progress panel renders after the catalogue instead of before it.

---

## Technical Context

**Language/Version**: TypeScript 5.8, strict mode

**Primary Dependencies**:
- `next@^15.3.4` — App Router nested layouts, `usePathname()` for active-link state
- `react@^19.0.0` — `useState`, `useEffect` (Escape-key listener, drawer close-on-route-change)
- `tailwindcss@^3.4.0` — responsive sidebar/drawer, `order-*` utilities for the mobile reorder fix

**No new npm packages** — no drawer/menu library added; the overlay drawer is a plain fixed-position
div with a backdrop, matching Constitution Principle V (Simplicity Over Abstraction).

**Storage**: N/A — no persisted data introduced. Nav open/closed state is local, transient UI state.

**Testing**: Jest is not applicable to this feature — no pure calculation functions are introduced
(Constitution's Jest mandate is scoped to `lib/metrics.ts`). Verification is manual, via the
`quickstart.md` checklist (desktop sidebar, mobile drawer, routing, mobile reorder).

**Target Platform**: Browser (client components: nav toggle/drawer) + Next.js Server Components
(the new `dashboard/layout.tsx` and `dashboard/squads/page.tsx` route shell)

**Performance Goals**:
- SC-001/SC-002: Section switch and drawer open/close are synchronous client interactions
  (Next `<Link>` + local `useState`) — no perceptible delay.

**Constraints**:
- Reuses the existing `lg` (1024px) Tailwind breakpoint already used by the Catálogo two-column
  layout, so sidebar-vs-drawer and squad-panel-reorder activate at the same width (per spec Assumptions).
- Nav open/close state MUST stay local client UI state (`useState`), not squad Context/useReducer —
  Constitution Principle II scopes Context+useReducer specifically to squad composition, not
  arbitrary UI state.
- The mobile reorder fix MUST be CSS-only (`order-*`), not a DOM/component restructure, to avoid
  touching `SquadPanel`/`CatalogueView` internals or their existing tests.
- Type `any` prohibited — strict TypeScript throughout.

**Scale/Scope**: 2 nav destinations, 1 new route (`/dashboard/squads`), 1 new layout, 1 new nav
component, 1 CSS reorder fix on the existing Catálogo page. No new backend/API surface.

added as `app/(private)/dashboard/layout.tsx` so it automatically wraps every route nested under
`dashboard/` (today: `page.tsx` and the new `squads/page.tsx`), which is the minimal way to share
navigation across both screens without duplicating markup or introducing a new state manager.
