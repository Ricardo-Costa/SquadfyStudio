# Implementation Plan: Server-Side Authentication

**Branch**: `001-server-action-auth` | **Date**: 2026-06-29 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/001-server-action-auth/spec.md`

## Summary

Implement a secure, server-side authentication flow for Squadfy Studio. A Tech Lead logs
in via a form that submits to a Next.js Server Action. On success, a signed JWT is stored
in an HttpOnly cookie and the user is redirected to `/dashboard`. Next.js middleware
validates the cookie on every protected-route request and redirects unauthenticated users
to `/login`. Invalid login attempts are rate-limited (5 attempts per IP → 15-minute
lockout, in-memory). The login page is visually polished with a loading state during
Server Action processing, setting the UI quality baseline for all subsequent features.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode)

**Primary Dependencies**:
- Next.js 15+ (App Router, Server Actions, Middleware — Edge Runtime)
- `jose` ^5.x — JWT signing and verification (Edge-compatible; `jsonwebtoken` rejected
  because it fails in Edge Runtime — see research.md Decision 1)
- Tailwind CSS — login page styling
- React 19 `useFormStatus` — loading state for Server Action form (built-in, no extra dep)

**Storage**: HttpOnly cookie (session token) + module-level `Map` in Node.js process (rate limiting)

**Testing**: No Jest tests for this feature — auth utilities (`lib/auth.ts`, `lib/rate-limit.ts`)
are pure functions but not in scope for unit tests per constitution (Jest covers `lib/metrics.ts` only).
Manual validation via quickstart.md.

**Target Platform**: Web browser (client UI) + Node.js runtime (Server Action) + Edge Runtime (middleware)

**Performance Goals**: Login flow < 2s (SC-001); error feedback < 1s (SC-004); loading
indicator visible within 100ms of submit (SC-007)

**Constraints**:
- Middleware MUST run on Edge Runtime — no Node.js APIs in `middleware.ts`
- Rate limiting MUST run in Node.js Server Action — not middleware (see research.md Decision 2)
- No external auth service, no database
- `AUTH_SECRET` min 32 chars for HS256 security margin

**Scale/Scope**: Single fixed-credential user; no concurrency beyond rate limiting counter

```text

  (auth)/
    login/
      page.tsx                    # Server Component — renders LoginForm
      actions.ts                  # Server Action: login()
      _components/
        LoginForm.tsx             # 'use client' — useActionState wires form to action
        LoginInput.tsx            # 'use client' — reusable input with label + error state
        SubmitButton.tsx          # 'use client' — useFormStatus for disabled/loading state

  (private)/
    dashboard/
      page.tsx                    # Protected page (placeholder for feature 002)

lib/
  auth.ts                         # signToken(), verifyToken(), TokenPayload type
  rate-limit.ts                   # checkRateLimit(), recordFailedAttempt(), resetAttempts()

middleware.ts                     # Edge Runtime — cookie validation + redirect logic

.env.local                        # AUTH_EMAIL, AUTH_PASSWORD, AUTH_SECRET (gitignored)
.env.example                      # Safe template with placeholder values (committed)
```

**Structure Decision**: Next.js App Router with route groups — `(auth)` for public routes,
`(private)` for protected routes. Login-specific components use `_components/` colocation
to keep them out of the router and signal they are not shared across features.
